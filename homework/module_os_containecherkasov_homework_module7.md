## 1. Мета роботи

Мета цього завдання — показати, що Docker-контейнер є звичайним Linux-процесом із ізоляцією.  
Контейнер має власне середовище, мережу, файлову систему та головний процес, який працює як `PID 1` усередині контейнера.

---

## 2. Запуск контейнера з простим застосунком

Для прикладу використовується Python HTTP-сервер.

Команда запуску контейнера:

```bash
docker run -d --name python-web -p 8080:8000 python:3.12-slim python -m http.server 8000
```

Пояснення команди:

```text
-d                      запуск у фоновому режимі
--name python-web       ім’я контейнера
-p 8080:8000            порт 8080 на хості перенаправляється на порт 8000 у контейнері
python:3.12-slim        Docker-образ
python -m http.server   запуск простого HTTP-сервера
8000                    порт усередині контейнера
```

Перевірка, що контейнер запущений:

```bash
docker ps
```

Очікуваний результат:

```text
CONTAINER ID   IMAGE              COMMAND                  PORTS                    NAMES
xxxxxxxxxxxx   python:3.12-slim    "python -m http.ser…"    0.0.0.0:8080->8000/tcp   python-web
```

Короткий коментар:

Контейнер запущено у фоновому режимі.  
На хості використовується порт `8080`, а всередині контейнера застосунок слухає порт `8000`.

---

## 3. Перевірка доступності HTTP-сервера

Команда:

```bash
curl http://localhost:8080
```

Або в браузері:

```text
http://localhost:8080
```

Короткий коментар:

Якщо сервер працює, у відповідь буде отримано HTML-сторінку зі списком файлів каталогу.  
Це означає, що port mapping між хостом і контейнером працює.

---

## 4. Аналіз процесів усередині контейнера

Щоб подивитися процеси всередині контейнера, використовується команда:

```bash
docker exec -it python-web ps aux
```

Якщо команди `ps` немає в образі, можна використати:

```bash
docker exec -it python-web sh -c "cat /proc/1/cmdline"
```

Очікуваний результат для `ps aux`:

```text
USER       PID %CPU %MEM    VSZ   RSS TTY      STAT START   TIME COMMAND
root         1  0.0  0.0  ...    ... ?        Ss   ...     0:00 python -m http.server 8000
```

Короткий коментар:

Процес `python -m http.server 8000` має `PID 1` усередині контейнера.  
Це головний процес контейнера, тому саме він визначає життя контейнера.

---

## 5. Чому саме цей процес є PID 1

Docker запускає команду, передану після назви образу:

```bash
python -m http.server 8000
```

Саме ця команда стала головним процесом контейнера.  
У звичайній Linux-системі `PID 1` часто має процес `systemd`, але в контейнері `PID 1` — це зазвичай сам застосунок.

Короткий коментар:

Контейнер не запускає повну операційну систему.  
Він запускає один основний процес у ізольованому середовищі.

## 6. Перегляд логів контейнера

Команда:

```bash
docker logs python-web
```

Щоб створити запис у логах, можна виконати запит:

```bash
curl http://localhost:8080
```

Потім знову переглянути логи:

```bash
docker logs python-web
```

Приклад результату:

```text
172.17.0.1 - - [27/Jun/2026 19:30:00] "GET / HTTP/1.1" 200 -
```

Короткий коментар:

`docker logs` показує те, що процес контейнера виводить у `stdout` та `stderr`.  
Тобто Docker не створює логи сам із повітря, а збирає стандартний вивід головного процесу.

## 7. Зупинка контейнера через docker stop

Команда:

```bash
docker stop python-web
```

Перевірка:

```bash
docker ps
```

Щоб побачити зупинені контейнери:

```bash
docker ps -a
```

Короткий коментар:

Команда `docker stop` надсилає головному процесу контейнера сигнал `SIGTERM`.  
Якщо процес завершується після цього сигналу, контейнер також завершується.

---

## 8. Що відбувається при docker stop

Коли виконується команда:

```bash
docker stop python-web
```

Docker спочатку надсилає процесу `PID 1` сигнал:

```text
SIGTERM
```

Цей сигнал означає прохання коректно завершити роботу.  
Якщо процес не завершується за відведений час, Docker надсилає:

```text
SIGKILL
```

`SIGKILL` примусово завершує процес.

Короткий коментар:

Контейнер завершується не тому, що “вимкнулась маленька віртуальна машина”.  
Він завершується тому, що завершився його головний процес.

---

## 9. Що буде, якщо процес ігнорує SIGTERM

Якщо процес усередині контейнера ігнорує сигнал `SIGTERM`, Docker певний час чекає.  
Після цього Docker надсилає сигнал `SIGKILL`, який процес не може проігнорувати.

Приклад команди з коротшим очікуванням:

```bash
docker stop -t 3 python-web
```

Пояснення:

```text
-t 3 означає, що Docker чекатиме 3 секунди перед примусовим завершенням.
```

Короткий коментар:

`SIGTERM` дає процесу шанс завершитися нормально.  
`SIGKILL` завершує процес примусово, без можливості коректно закрити файли або зберегти стан.

---

## 10. Повторний запуск і перевірка

Якщо контейнер був зупинений, його можна запустити знову:

```bash
docker start python-web
```

Перевірити статус:

```bash
docker ps
```

Переглянути процеси:

```bash
docker exec -it python-web ps aux
```

Короткий коментар:

Після повторного запуску всередині контейнера знову запускається головний процес `python -m http.server 8000`.

---

## 11. Очищення після виконання роботи

Зупинка контейнера:

```bash
docker stop python-web
```

Видалення контейнера:

```bash
docker rm python-web
```

Короткий коментар:

Після завершення роботи контейнер можна видалити, щоб він не займав місце в системі.

---

## 12. Висновок

Контейнер — це не повноцінна віртуальна машина, а ізольований Linux-процес.  
У цьому завданні головним процесом контейнера був `python -m http.server 8000`, який мав `PID 1` усередині контейнера.

Життя контейнера визначається життям його головного процесу.  
Коли цей процес завершується або отримує сигнал завершення, контейнер також зупиняється.

Логи контейнера беруться зі стандартних потоків `stdout` та `stderr`.  
Команда `docker logs` просто показує вивід, який створив процес усередині контейнера.
